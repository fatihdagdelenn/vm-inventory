VM Envanter Yönetim Sistemi

VMware vCenter ve Proxmox VE ortamlarındaki tüm sanal makineleri, hostları ve ağları tek web arayüzünden izleyen merkezi envanter. Bu doküman, sistemi sıfırdan kuracak biri için tüm adımları sırasıyla anlatır.

Python 3.11+ · FastAPI PostgreSQL 16 Docker Compose vCenter 6.7 – 8.x Proxmox 7.x / 8.x Türkçe arayüz

Ne işe yarar?

Birden fazla sanallaştırma ortamı işleten ekiplerin günlük sorularını tek arama kutusunda yanıtlar: "10.10.10.15 hangi host'ta?", "VLAN 100'de kaç Windows sunucu var?", "geçen hafta hangi VM'lerin RAM'i değişti?", "agent kurulu olmayan VM'ler hangileri?"

SSH erişimi veya misafire ajan dağıtımı gerekmez — yalnızca platformların resmî API'leri kullanılır ve salt-okunur hesaplar yeterlidir. Sistem hiçbir VM'e müdahale edemez; bilinçli olarak salt-okunur tasarlanmıştır.

Mimari: nasıl çalışır?

┌─────────────┐ zamanlanmış görevler ┌──────────────────┐ │ vCenter(s) │◄──────(APScheduler)────────┤ │ └─────────────┘ 15 dk'da bir, │ FastAPI │ ┌─────────────┐ pyVmomi/proxmoxer │ + SQLAlchemy │──► PostgreSQL │ Proxmox(es) │◄───────────────────────────┤ │ (lokal cache) └─────────────┘ └────────┬─────────┘ │ tüm aramalar lokal DB'den ┌──────▼─────────┐ │ Web Arayüzü │ └────────────────┘

Temel ilke: kullanıcı aramaları asla canlı API'ye gitmez.

  1. Toplayıcı — Arka plan zamanlayıcısı her 15 dakikada (ayarlanabilir) platformlara bağlanır. vCenter'da ContainerView ile tek seferde toplu okuma, Proxmox'ta tek cluster/resources çağrısı yapılır — host/VM başına ayrı istek atılmaz, canlı ortama yük binmez.
  2. Cache — VM adı, IP, MAC, OS, CPU/RAM/disk, VLAN, cluster, güç durumu, Tools/Agent durumu indeksli tablolara yazılır; farklar Değişiklik Geçmişi'ne işlenir.
  3. Arayüz — Aramalar, raporlar, grafikler lokal veritabanından sunulur; 500+ VM'de dahi anlıktır. "Tümünü Yenile" butonu acil durumda anında senkronizasyon tetikler.

ADIM 1 Docker ile Kurulum (önerilen)

  1. Projeyi indirin

    git clone <repo-adresi> vm-inventory
cd vm-inventory

  2. Ortam dosyasını oluşturun

    cp .env.example .env

  3. Güvenlik anahtarlarını üretin

    İki anahtar gerekir; komutların çıktısını .env içine yapıştırın:

    # ENCRYPTION_KEY — platform parolalarını/token'larını şifreler:
python3 -c "import base64, os; print(base64.urlsafe_b64encode(os.urandom(32)).decode())"

# SECRET_KEY — oturum çerezlerini imzalar:
python3 -c "import secrets; print(secrets.token_urlsafe(48))"
    SECRET_KEY=ürettiğiniz-secret-key
ENCRYPTION_KEY=ürettiğiniz-fernet-anahtarı
SYNC_INTERVAL_MINUTES=15
USAGE_SYNC_INTERVAL_MINUTES=3
SESSION_TIMEOUT_MINUTES=480
    ⚠️
    ENCRYPTION_KEY'i yedekleyin. Kaybedilirse kayıtlı platform kimlik bilgileri çözülemez ve yeniden girilmesi gerekir.

  4. (İsteğe bağlı) Portu değiştirin

    docker-compose.yml içindeki app servisinde yalnızca soldaki sayıyı değiştirin:

    ports:
  - "18443:8000"   # dış port : konteyner portu

    Portun boş olduğunu kontrol etmek için: ss -tlnp | grep 18443

  5. Başlatın

    docker compose up -d --build

    İlk build 2-3 dakika sürer. İzlemek için: docker compose logs -f app

  6. Giriş yapın

    Tarayıcıda http://sunucu-adresi:8000 (veya seçtiğiniz port) açın.

    🔑
    Varsayılan giriş: admin / admin123 — ilk iş olarak değiştirin (Adım 3).

ADIM 2 Manuel Kurulum (Docker'sız alternatif)

Docker kullanamıyorsanız:

# Python 3.11+ ve venv gerekir (sistem paketleriyle çakışmayı önler)
python3 -m venv venv && source venv/bin/activate
pip install -r requirements.txt

cp .env.example .env        # anahtarları Adım 1.3'teki gibi üretin
mkdir -p data/reports

uvicorn app.main:app --host 0.0.0.0 --port 8000

Küçük kurulumlar için SQLite yeterlidir — .env içinde: DATABASE_URL=sqlite:///./data/vminventory.db

PostgreSQL tercih ederseniz önce veritabanını hazırlayın:

CREATE USER vminv WITH PASSWORD 'guclu_parola';
CREATE DATABASE vminventory OWNER vminv;

ve .env'de: DATABASE_URL=postgresql+psycopg2://vminv:guclu_parola@localhost:5432/vminventory

💡
Tablolar uygulamanın ilk açılışında otomatik oluşturulur; migration komutu gerekmez.

ADIM 3 İlk Yapılandırma

ADIM 5 Proxmox Bağlama

5.1 — API token üretin

Proxmox node'unda (SSH veya web arayüzündeki Shell) üç komut:

pveum user add svc-envanter@pve --comment "Envanter servis hesabı"
pveum acl modify / --users svc-envanter@pve --roles PVEAuditor
pveum user token add svc-envanter@pve envanter --privsep 0
⚠️
Son komutun çıktısındaki value satırını hemen kaydedin — bir daha asla gösterilmez.
⚠️
--privsep 0 zorunludur. Verilmezse token VM'leri göremez (sadece node adları gelir). Yanlışlıkla atlandıysa düzeltme: pveum acl modify / --tokens 'svc-envanter@pve!envanter' --roles PVEAuditor (tırnaklar önemli — ! kabukta özel karakterdir).

ADIM 6 İlk Senkronizasyon

  1. Platform kartındaki "Senkronize Et" butonuna basın. Arka planda çalışır, arayüz kilitlenmez; ortam büyüklüğüne göre 1-5 dakika sürer.
  2. Karttaki durum "Bağlantı OK" olunca Dashboard ve Sanal Makineler ekranı dolmuş demektir. Hata olursa karttaki Loglar butonu API hatasının tam mesajını gösterir.
  3. Sonrası otomatiktir: her 15 dakikada bir tazelenir, farklar Değişiklik Geçmişi'ne işlenir.
💡
Önerilen son adım: Dashboard'daki "Dikkat Gerektirenler" kartından Agent/Tools kurulu olmayan listesine bakın. Bu VM'lerin IP'si ve gerçek OS adı alınamaz. vCenter VM'lerine VMware Tools, Proxmox VM'lerine apt install qemu-guest-agent (+ VM Options'ta agent'ı etkinleştirip yeniden başlatma) kurulduğunda bir sonraki senkronizasyonda "Ubuntu 22.04.3 LTS" gibi tam OS adları ve IP'ler gelir.

Cluster Yönetimi & Anlık Kullanım

Cluster göster/gizle

Dashboard'da "Cluster Bazında VM Dağılımı" grafiğinin başlığındaki Yönet butonu, tüm cluster'ları VM/host sayılarıyla listeleyen bir pencere açar. Anahtarı kapattığınız cluster:

  • Dashboard'daki tüm sayılara ve grafiklere dahil edilmez
  • VM listesinde varsayılan olarak görünmez
  • Veri silinmez — gelişmiş paneldeki "gizli cluster'ları dahil et" kutusu veya doğrudan cluster: araması yine erişir; anahtarı açınca her şey geri gelir
ℹ️
Listenin başındaki (Cluster'sız) sanal grubu, herhangi bir cluster'a bağlı olmayan (standalone host'lardaki) VM ve host'ları temsil eder — grafiklerde "—" olarak görünen kayıtlar bunlardır. Bu grup da aynı anahtarla gizlenebilir; cluster:yok aramasıyla her zaman listelenebilir.

Anlık CPU / RAM / disk kullanımı

Envanter senkronizasyonundan bağımsız, çok hafif bir görev her USAGE_SYNC_INTERVAL_MINUTES dakikada bir (varsayılan 3) ve her uygulama açılışından ~20 saniye sonra çalışır; Proxmox'ta tek cluster/resources çağrısı, vCenter'da quickStats okumasıyla biter — canlı ortama yük bindirmez.

  • VM tablosunda CPU/RAM/disk kolonlarının altında mini kullanım çubukları (sarı %75+, kırmızı %90+); üzerine gelince tam değer görünür
  • Disk için vCenter'da gerçek doluluk gösterilir (thin-provisioned diskte tahsis edilen değil, datastore'da fiilen yazılı alan); Proxmox'ta agent'lı VM'lerde gelir
  • Kapalı VM'lerde çubuk çizilmez — anlık kullanım yoktur
  • Dashboard'daki host grafikleri aynı tazeliktedir; başlıkta son tazeleme zamanı görünür

LDAP kullanıcıları ilk girişte otomatik oluşturulur; rolleri Yönetim ekranından yükseltilir. Lokal admin hesabı LDAP'tan bağımsız çalışmaya devam eder.

Raporlama

Anlık dışa aktarma: VM listesinde mevcut filtre neyse, Excel/CSV/PDF butonları o filtrelenmiş sonucu indirir. Raporlar ekranından da filtre yazıp export alabilirsiniz.

Zamanlanmış raporlar: Raporlar ekranında saat, format ve filtre belirleyin; sistem her gün o saatte raporu üretip sunucudaki data/reports/ klasörüne (Docker'da appdata volume'ü) yazar. Örnek: env:production filtresiyle yalnızca production VM'leri içeren günlük Excel raporu.

Üretim Ortamı Notları

HTTPS

Uygulamanın önüne reverse proxy koyun (nginx örneği):

server {
    listen 443 ssl;
    server_name envanter.sirket.local;
    ssl_certificate     /etc/ssl/certs/envanter.crt;
    ssl_certificate_key /etc/ssl/private/envanter.key;
    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto https;
    }
}

Yedekleme

İki şey yeterli: veritabanı ve .env dosyası (özellikle ENCRYPTION_KEY):

docker compose exec db pg_dump -U vminv vminventory > yedek_$(date +%F).sql

Güncelleme

# yeni sürüm dosyalarını proje dizinine açtıktan sonra:
docker compose up -d --build
# tarayıcıda Ctrl+F5 (eski CSS/JS önbelleğini atmak için)

Veriler volume'lerde korunur, tablolar gerekirse otomatik genişletilir.

SSS

Canlı ortama yük bindirir mi? Hayır. 15 dakikada bir, toplu-okuma API çağrıları yapılır. Kullanıcı trafiği platformlara hiç ulaşmaz.

Veriler ne kadar güncel? En fazla SYNC_INTERVAL_MINUTES kadar geride (varsayılan 15 dk). "Tümünü Yenile" ile anında tazelenir.

Kaç platform eklenebilir? Sınır yok; hepsi aynı envanterde birleşir.

VM'lere müdahale edebilir mi (başlat/durdur)? Hayır — bilinçli tasarım kararı. Sistem salt-okunur çalışır; bu sayede platform hesapları da salt-okunur olabilir.

SQLite mi PostgreSQL mi? Tek kullanıcılı küçük kurulumda SQLite yeterli; ekip kullanımı ve 500+ VM için PostgreSQL önerilir (Docker Compose varsayılanı).
